# 3.10 & 3.x — Finding Issues Early & Chapter Summary

The best way to debug is to avoid making errors in the first place or catching them immediately.

- **The Debugging Process:** Finding a semantic error (where valid syntax does the wrong thing) is usually the hardest part. The five steps to debugging are: find the root cause, understand the problem, determine a fix, repair the issue, and retest.
    
- **Refactoring:** Making structural changes to your code to increase organization and modularity without changing its behavior. Functions should ideally be under ten lines to maximize comprehension.
    
- **Defensive Programming:** Anticipating all the ways the software could be misused by end-users or other developers.
    
- **Unit Testing:** A testing method where small units of source code are tested to determine if they are correct.
    
- **Constraints:** Adding extra code (like `assert` or `static_assert`) to check that assumptions are not violated. These can be compiled out in non-debug builds.
    
- **Static Analysis (Linters):** Tools that analyze your source code to identify specific semantic issues without executing the code. Your `tasks.json` is already highly optimized for this, using strict compiler flags like `-Wall`, `-Wextra`, and `-pedantic-errors` to act as a powerful static analysis tool.
